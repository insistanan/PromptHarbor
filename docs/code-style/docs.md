# 文档写作规范

## 语言

所有项目文档默认使用中文。术语首次出现时可以保留英文原词，例如 hook、provider、session、prompt、spool。

## 结构

文档应先说明背景和目标，再说明方案、约束、风险和待验证项。

推荐结构：

- 背景。
- 目标与非目标。
- 设计方案。
- 数据或接口模型。
- 风险与取舍。
- 里程碑或待办。
- 参考资料。

不要求每份文档都套模板，但要让读者能快速判断“为什么做、怎么做、暂时不做什么”。

## 内容粒度

- 根级 `AGENTS.md` 和 `CLAUDE.md` 只放协作入口和索引。
- 详细代码规范放在 `docs/code-style/`。
- 产品和架构方案放在 `docs/`。
- 未来如果有架构决策记录，建议放在 `docs/adr/`。

## 写作风格

- 使用直接、明确的表达，不写空泛口号。
- 能列约束就列约束，能列验收标准就列验收标准。
- 对尚未实测的内容标记为“待验证”，不要写成已确定事实。
- 对外部工具行为要注明来源或验证方式。

## 代码块

- 命令示例要标注 shell 类型，例如 `powershell`、`toml`、`json`、`sql`。
- Windows 路径示例要注意反斜杠转义。
- 配置示例应尽量最小化，不复制无关字段。

## 更新要求

- 修改设计方向时，同步检查 `docs/promptbox-design.md` 是否需要更新。
- 修改代码规范入口时，同步检查 `AGENTS.md` 和 `CLAUDE.md`。
- 如果文档变长导致编辑失败，应分批编辑。
