# 前端代码规范

## 技术边界

前端以 Tauri + React + Milkdown 为基础：

- Tauri 负责桌面壳、本地能力和 Rust 后端通信。
- React 负责界面状态和组件组织。
- Milkdown 负责 Markdown 所见即所得编辑。
- SQLite 数据不直接暴露给前端，统一通过 Tauri command 或本地后端 API 访问。

## 信息架构

第一版界面应围绕三个区域组织：

- 会话列表：按项目、provider、更新时间和搜索条件筛选。
- prompt 历史：只展示用户发送过的 prompt。
- 草稿编辑器：当前会话的 Milkdown 草稿。

不要把第一屏做成营销页或说明页。打开应用后应直接进入可编辑、可检索的工作界面。

## Milkdown 使用

- 使用 Milkdown 管理编辑体验，但保存时以 Markdown 源文本为准。
- 使用 listener 监听 Markdown 变化，并做 debounce 自动保存。
- 使用 `getMarkdown()` 或等价方式获取当前 Markdown 文本。
- 不在编辑器插件里混入业务持久化逻辑；保存逻辑放在 React 状态层或应用服务层。
- 第一版避免复杂自定义插件，优先保证稳定编辑、复制和恢复草稿。

## 状态管理

- 会话、prompt 历史、草稿内容要分开建模。
- 已发送 prompt 以 hook 捕获结果为准，不以草稿内容推断。
- 草稿保存失败时要保留前端内容，并给出清晰状态。
- 避免全局状态过早复杂化；第一版可以使用 React Query 或轻量 store。

## UI 规范

- 界面应偏工具型、信息密度适中，避免装饰性过强。
- 按钮、筛选、复制、删除等操作要有明确反馈。
- 删除会话或 prompt 前要二次确认。
- 长 prompt 要支持折叠、展开和复制。
- provider 使用清晰文本或图标标识，不依赖颜色作为唯一线索。
- 不使用大面积单一色系渐变背景。

## 快捷操作

第一版推荐支持：

- 复制当前草稿。
- 清空当前草稿。
- 从历史 prompt 插入到草稿。
- 搜索 prompt。

自动注入终端输入框暂不作为第一版能力，避免焦点、终端类型和 TUI 行为带来的不稳定。

## 与后端通信

- 前端不要直接拼 SQL。
- Tauri command 返回值要使用明确类型。
- 错误返回要包含用户可读信息和内部错误码。
- 大文本传输要考虑大小限制和失败提示。

## 可访问性

- 关键操作支持键盘触达。
- 输入区域和按钮焦点状态要明显。
- 文本对比度要满足长时间阅读需求。
- 窗口较小时不能出现关键按钮被挤出或文本重叠。
